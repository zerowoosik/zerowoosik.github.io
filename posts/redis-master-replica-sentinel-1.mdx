## 1. 서버 사양 및 구성 구상하기

현재 사용하는 서버는 Oracle cloud를 그대로 이용하고 있다. OKE를 통해 쿠버네티스 환경을 구성하게 되면 문제는 ARM A1을 모두 사용하게 되어 사실상 Redis를 돌릴만한 사양의 무료 인스턴스를 모두 소모한 셈이 된다. 그래서 가능하다면 계정을 하나 더 만들어 Redis서버를 구축하기로 했다. 서로 다른 아이디(테넌시)인 경우 서버가 분리되지만 피어링을 통해서 연결하여 사용할 예정이다. 이 내용은 다음 포스팅에서 다룬다.

공식 문서에 따르면 개발 환경에는 최소 1노드, 권장 2노드 이상 요구하고 있으며, 노드별 최소 4GB 이상의 램, 권장 10GB 이상의 램을 요구한다.(참조 : [Redis Docs](https://redis.io/docs/latest/operate/rs/installing-upgrading/install/plan-deployment/hardware-requirements/))&#x20;

저기에 부합하는 가장 좋은 OCI 인스턴스는 ARM A1 2CPU 12GB RAM 사양을 선택하는것이 가장 최적의 조건이다. 두 개의 인스턴스를 만들어 Master-replica 구성을 두고, 1개의 AMD 1CPU 1GB RAM 인스턴스를 만들어 sentinel을 추가하면 무료 사양에서 가장 최적의 인스턴스를 만들 수 있다. 표로 나타내면 다음과 같다

|                            | redis-server | redis-sentinel |
| -------------------------- | ------------ | -------------- |
| SERVER 1 : A1 2CPU 12GB    | Master       | O              |
| SERVER 2 : A1 2CPU 12GB    | replica      | O              |
| SERVER 3 : AMD 1 OCPU, 1GB | X            | O              |

레디스 서버가 동작하지 않는 AMD 인스턴스는 단순 sentinel기능만 하게 하여 quroum의 수가 홀수개가 유지되도록 설정하는 목적이기에 실질적으로 큰 사양을 필요로 하지 않는다. 게다가 Redis는 공식적으로 ARM, X86 모두 지원하므로 구성에 큰 어려움도 없다.

서비스를 구성하기 위해 인스턴스를 생성하자.



## 2. 인스턴스 생성하기

여느때와 같이 인스턴스는 위 사양대로 생성하면 된다. 다만, 레디스의 master-replica 구성 및 센티넬은 각자 다른 서버에서 구동되어야 한다. 보통 AWS에서 AZ(Availability Zone)으로 나누듯이 OCI에서도 나눠야 하는데 OCI의 용어는 AD(Availability Domain)이다. (출처 : [OCI Region & AD](https://docs.oracle.com/en-us/iaas/Content/General/Concepts/regions.htm))

리스트에서 보듯 3개 이상의 AD를 가지고 있는 리전은 몇 개 없다. 그러면 서버를 어떻게 나눠야 할 것인가? 바로 FD(Fault Domains)를 이용하면 된다. 위 출처 설명에도 나와있지만 FD를 이용하면 물리적으로 인스턴스를 분리시켜 장애 발생점을 나누는 방법이라고 보면 된다.

![](/images/redis-master-replica-sentinel-1/1.png)

인스턴스를 생성할 때 `이미지 및 구성` 위에 `고급 옵션`을 누르면 아래 결함 도메인을 선택할 수 있다. 서버 1, 2, 3번에 각각 결함 도메인을 1,2,3으로 매칭해 줄 것이다.

![](/images/redis-master-replica-sentinel-1/2.png)

Oracle cloud는 어차피 200GB의 부트 볼륨을 제공해주고있다. Redis는 RDB, AOF를 이용한 영속성을 제공하기 위해 디스크도 사용하고 있기 때문에 가능하면 다 사용하는게 좋다. 어차피 3개 서버를 사용할 예정이고 인스턴스 하나당 최소 부트 볼륨은 50GB를 설정해야하니 Server1,2에는 75GB, Server3에는 50GB를 할당해준다.

VPU가 늘어나면 비용이 부과될 수 있기 때문에 VPU를 늘리지 말고 볼륨 크기를 늘려서 처리량의 속도를 늘리는게 그나마 현명한 방법이다.

![](/images/redis-master-replica-sentinel-1/3.png)

![](/images/redis-master-replica-sentinel-1/4.png)

추가적으로 추후에 혹시나 모를 상황에 대비하여 좀 더 빠르게 동일 환경을 추가하기 위해 이번엔 OCI에서 제공하는 terraform을 이용해본다. 설정을 모두 마치고 인스턴스 생성을 누르지 말고 `스택으로 저장` 을 눌러서 별 다른 수정없이 next를 눌러 설정을 완료하자. 마지막에 적용 실행만 진행하면 인스턴스가 생성되면서 스택에 terraform이 저장된다. terraform은 무료로 제공된다고 되어있으니 편하게 사용하면 될 듯 하다.

![](/images/redis-master-replica-sentinel-1/5.png)

생성된 스택의 패널에 보면 terraform 구성을 다운로드 받을 수 있다. Server 1과 동일한 서버를 만들 예정이니 해당 terraform을 다운로드 받아서 최소 다음 세 가지 요소를 수정하자

```tf
create_vnic_details - display_name
display_name
fault_domain

metadata - ssh_authorized_keys // 필요한 key를 만들어서 public 키를 넣으면 된다. 가능하면 구분하는게 좋음
```

디스플레이 네임은 개개인 마다 가지고있는 네이밍 규칙에 맞추고, fault domain은 2로 수정해서 저장 후 압축한다. 그리고 OCI의 `스택` 페이지를 검색해서 들어간 뒤에 `스택 생성`을 눌러 zip파일 첨부 후 next 진행하면 된다. 같은 사양의 인스턴스가 뚝딱 나온다.

AMD 인스턴스는 그냥 새로 만들어주고, FD만 3번으로 따로 설정해주면 된다. 참고로 내부망으로 사용할거니 공인 IP는 따로 필요없다.



## 3. 외부 접속 설정하기

인스턴스가 모두 만들어졌다면 외부에서 최소한 ssh 접속은 가능해야한다. ssh 접속을 하는 방법은 여러가지가 있는데 보통 나는 두 가지를 사용한다.

1. cloud shell 을 이용해 tailscale 설치 후 접속
2. bastion service로 접속

1번의 경우 cloud shell에 접속 키를 주고 이런 저런 귀찮은게 한 번 필요하지만, tailscale을 설치해두면 TTL없이 항상 접속할 수 있다는 장점이 있다.

2번의 경우는 bastion 서비스를 이용해서 좀 더 보안성있는 접속이 가능하고 TTL이 있기 때문에 일정 시간이 지나면 세션을 자동으로 삭제해주기 때문에 안전하다는 장점이 있다.(여긴 기본 TTL이 3시간인가보다) 나는 어차피 큰 문제가 없다면 한 번 구성해놓고 따로 손을 대지 않을 것이니 bastion을 이용해보기로 했다.

![](/images/redis-master-replica-sentinel-1/6.png)

사실 bastion을 이용하려면 terraform에서 agent\_config - plugins\_config 중 BASTION 옵션을 enable 해야한다. 처음에 안 하고 생성하면 인스턴스에 들어가서 상단의 관리 - Oracle Cloud 에이전트 항목에서 `배스천`을 사용으로 변경하면 된다.

서비스 - 배스천에 들어가서 배스천을 생성해준다. CIDR은 0.0.0.0/0 으로 설정.

![](/images/redis-master-replica-sentinel-1/7.png)

배스천 이름 적당히 설정해주고, 호스트 접속은 인스턴스로 설정, 포트 22는 그대로 두고 SSH키는 인스턴스 생성시 만들거나 받은 키의 pub 파일 넣어주면 된다. 세션이 생성되면 생성 리스트 우측 점 세개 옵션을 클릭하면 `SSH 명령 복사` 버튼을 누르면 SSH 터널링 명령어를 준다. 해당 명령어를 통해 터널링하여 접속하면 된다. 세 개 모두 만들어주고 각 서버별로 로컬 포트를 2221, 2222, 2223 이런 식으로 사용했다.



### – NAT Gateway 설정 (간략 설명)

private subnet을 이용해서 인스턴스를 만든 경우 공인 IP가 할당되지 않기 때문에 NAT Gateway를 할당해주어야 한다. 그렇지 않으면 외부 통신이 막혀 패키지 설치 자체가 불가능하다. 간략히 설명한다.

1. 가상 클라우드 네트워크 - vcn 선택 - 상단 게이트웨이 클릭
2. 하단 NAT 게이트웨이 메뉴에서 NAT 게이트웨이 생성 - 이름 및 임시 퍼블릭 IP 주소 선택(외부 통신이기에 굳이 고정 IP는 필요없다) - NAT 게이트웨이 생성 클릭
3. 인스턴스 - 네트워크 - 경로 테이블 클릭 - 경로 테이블 페이지 상단 경로규칙 - 경로 규칙 추가 - NAT 게이트웨이 추가 - CIDR : 0.0.0.0/0&#x20;

위 과정을 통해 NAT Gateway 설정이 가능하다.



## 4. Redis 설치하기

|                            | redis-server | redis-sentinel |
| -------------------------- | ------------ | -------------- |
| SERVER 1 : A1 2CPU 12GB    | O            | O              |
| SERVER 2 : A1 2CPU 12GB    | O            | O              |
| SERVER 3 : AMD 1 OCPU, 1GB | X            | O              |

이제 Redis를 각 서버에 설치해야한다. 위 표처럼 SERVER 1,2에는 redis-server와 redis-sentinel을 실행해두고, SERVER 3에는 redis-sentinel만 실행할 예정이다. 설치는 공식 문서를 참고한다. (출처 : [Redis install on Linux](https://redis.io/docs/latest/operate/oss_and_stack/install/install-stack/apt/))

```bash
sudo apt-get install lsb-release curl gpg
curl -fsSL https://packages.redis.io/gpg | sudo gpg --dearmor -o /usr/share/keyrings/redis-archive-keyring.gpg
sudo chmod 644 /usr/share/keyrings/redis-archive-keyring.gpg
echo "deb [signed-by=/usr/share/keyrings/redis-archive-keyring.gpg] https://packages.redis.io/deb $(lsb_release -cs) main" | sudo tee /etc/apt/sources.list.d/redis.list
sudo apt-get update

sudo apt-get install redis-server redis-sentinel redis-tools
```

모든 서버 다 위 명령어로 설치해준다. SERVER 3은 redis-sentinel만 실행한다 해도 실행에 관련한 설정파일 같은게 종속성이 있는 것으로 보인다. 셋 다 설치를 해주어야 에러가 나지 않는다.

### – 인스턴스 포트 오픈하기

oracle cloud에서는 ufw를 통해 포트 오픈하는게 오류가 있다는 말이 종종 있어 iptables를 이용해서 포트를 오픈하기로 한다.

```bash
sudo iptables -I INPUT 1 -p tcp --dport 6379 -j ACCEPT  # redis-server 포트
sudo iptables -I INPUT 1 -p tcp --dport 26379 -j ACCEPT  # redis-sentinel 포트
```

위 명령어를 각 서버에 모두 입력하면 내부적으로 VM 인스턴스 내에서 포트를 오픈할 수 있다.

Oracle cloud 콘솔에서도 Ingress 포트 오픈을 해줘야 한다. (간략히 설명한다.)

1. 인스턴스 - 네트워킹 - 서브넷 클릭
2. 서브넷 상단 보안 클릭 - Security List 클릭
3. Security List 의 상단 보안 규칙 클릭 - 수신 규칙(Ingress) 에서 22번 포트로 열린 규칙 편집 - 22번 포트 제거 후 저장(모든 포트 오픈)

(프라이빗 서브넷으로 만든 경우에는 모두 오픈해도 크게 문제가 없겠지만 프라이빗이 아닌 경우에는 필요한 포트만 오픈)



### – 각 서버별 세팅하기 (SERVER 1, 2)

```bash
sudo systemctl start redis-server
sudo systemctl start redis-sentinel

sudo systemctl enable redis-server
sudo systemctl enable redis-sentinel
```

SERVER 1,2는 서버와 센티넬 모두 사용하니까 재부팅이 되더라도 실행될 수 있게 설정해준다.

`sudo vi /etc/redis/redis.conf` 를 실행하여 다음과 같이 설정한다.

```bash
bind 0.0.0.0
protected-mode no

// 검색하면 내용이 나오니 그 부분 주석 해제 후 사용.
```

SERVER 1은 우선 마스터로 설정하고, SERVER 2는 레플리카로 설정할 예정이니 SERVER 2에는 다음 옵션도 활성화 해야한다.

```
replicaof <MASTER Internal IP> 6379
```

replica로 사용할 서버는 해당 값을 넣어야 한다. SERVER 1 의 내부망 IP 를 넣으면 된다.



### – 각 서버별 세팅하기 (SERVER 3)

```bash
sudo systemctl stop redis-server
sudo systemctl start redis-sentinel

sudo systemctl disable redis-server
sudo systemctl enable redis-sentinel
```

SERVER 3은 서버는 사용하지 않으니 종료 및 disable 해준다.



### – 공통 서버 세팅하기 (sentinel 세팅)

`sudo vi /etc/redis/sentinel.conf` 명령어 실행하여 센티넬에 대한 수정을 진행한다.

```
port 26379 # 기본값임

protected-mode no # 수정 필요

sentinel monitor mymaster <Master Node IP> 6379 2  # 2는 쿼럼의 최소 개수를 뜻함, 6379는 redis 포트
```



위 설정을 모든 서버에 적용하면 sentinel 세팅이 완료된다.&#x20;

```
sudo systemctl restart redis-server  # SERVER 3는 redis-server를 사용하지 않으니 사용하지 않아도 됨
sudo systemctl restart redis-sentinel
```



## 5. sentinel 설정 여부 테스트

위 방법대로 redis랑 sentinel이 모두 설정 완료되었다면 기본적으로 두 가지 체크가 필요하다.

1. **master-replica 세팅 및 sentinel 개수 확인**
2. **master node 종료 시 replica의 master 승격(failover) 확인**



```bash
$ redis-cli -p 26379 info sentinel

# Sentinel
sentinel_masters:1
sentinel_tilt:0
sentinel_tilt_since_seconds:-1
sentinel_total_tilt:0
sentinel_running_scripts:0
sentinel_scripts_queue_length:0
sentinel_simulate_failure_flags:0
master0:name=mymaster,status=ok,address=<Master Node IP>:6379,slaves=1,sentinels=3
```

위 명령어 입력을 통해서 현재 마스터의 개수, replica(slaves) 개수, sentinel 개수를 확인할 수 있다. 마스터 1개, 레플리카 1개씩 존재하면 되고, sentinels는 3개가 존재하면 구성은 우선적으로 성공이다.\
\
마스터 노드 인스턴스에서 `sudo systemctl stop redis-server` 를 입력한 뒤 `sudo tail -f /var/log/redis/redis-sentinel.log` 를 입력해서 로그를 지켜보면 마스터 노드의 상태가 종료된 것을 보고 failover 과정을 지켜볼 수 있다.

sentinel끼리 투표 후 replica를 master로 승격시키고, 위에서 수정한 redis.conf, sentinel.conf를 수정한 뒤 적용한다.\
`bash redis-cli -p 26379 info sentinel` 를 입력 해 보면 Master Node IP가 변경된 것을 확인할 수 있다.



다음은 VPS 피어링을 통한 다른 테넌시(다른 계정)와의 연결로 redis를 더 빠르게 사용하도록 설정해보겠다.
